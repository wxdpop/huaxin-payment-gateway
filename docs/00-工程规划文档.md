### 2.1 DDD 分层模型（单体+模块化）

```

│  │ 模块    │ 模块     │ 模块     │ 模块     │ 模块     │    │    (命名空间隔离)
│  └─────────┴──────────┴──────────┴──────────┴──────────┘    │
│  用例编排、事务边界、领域事件发布、模块间通过 应用服务/Kafka 通信 │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│              领域层 (PaymentGateway.Domain)                   │
│  Order聚合 │ Account聚合 │ Payment聚合 │ Refund聚合            │
│  实体、值对象、领域服务、领域事件、仓储接口(纯定义,零依赖)        │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│          基础设施层 (PaymentGateway.Infrastructure)           │
│  SqlSugar仓储 │ Redis锁 │ ZK锁 │ Kafka总线 │ ZeroMQ │ Jaeger    │
└─────────────────────────────────────────────────────────────┘
```

**学习要点**：单体+模块化的核心是"高内聚低耦合"——模块内部通过 DDD 聚合保持一致性，模块之间通过 应用服务（进程内同步直接调用）或 Kafka（异步事件）通信，避免直接依赖。这样未来拆分微服务时，只需将 进程内应用服务调用 替换为 HTTP/Kafka 调用，领域模型与基础设施不变。

### 2.2 业务模块划分

单体应用内划分 6 大业务模块，通过命名空间/文件夹隔离，每模块对应一个 DDD 聚合：

| 模块 | 职责 | 关键技术 | 对应聚合 |
|------|------|---------|---------|
| `Orders` | 订单管理：创建、状态机流转、幂等 | SqlSugar、状态机 | Order 聚合根 |
| `Routing` | 支付路由：根据金额/渠道/费率选择通道 | 策略模式、规则引擎 | - (领域服务) |
| `Payments` | 支付记录、回调处理 | Redis 锁、Kafka | PaymentRecord 聚合 |
| `Accounts` | 资金账户：余额增减、流水、对账 | ZK 互斥锁、乐观锁 | Account 聚合根 |
| `Channels` | 通道适配：微信/支付宝/银联对接 | ZeroMQ、Factory、Mock | - (防腐层) |
| `Notify` | 异步通知商户、重试补偿 | Kafka 消费者、Polly | - (应用服务) |

**模块间通信策略**：
- **同步调用**：应用服务 进程内请求/响应（如 Order 创建时调用 Routing 选择通道）
- **异步事件**：Kafka 事件总线（如回调处理完成后异步记账、通知商户）—— 学习 Kafka 异步解耦
- **低延迟通道通信**：ZeroMQ（Channels 模块内部模拟路由-通道通信，学习 ZeroMQ 模式）
- **分布式锁**：Redis-Cluster (Redlock) + ZooKeeper（Payments/Accounts 模块使用）

## 四、中间件设计封装

本节为学习工程的核心——所有中间件均提供接口抽象 + 实现类 + 详尽注释，便于理解原理与替换实现。

### 4.1 Redis-Cluster 分布式锁 (Redlock)

**设计目标**：以支付订单号为 Key 获取锁，防止多渠道重复回调导致重复入账。

**接口抽象**：
```csharp
public interface IDistributedLock
{
    Task<ILockHandle?> TryAcquireAsync(
        string resource,             // 资源Key(如订单号)
        TimeSpan expiryTime,         // 锁过期
        CancellationToken ct = default);
}

public interface ILockHandle : IAsyncDisposable
{
    string Resource { get; }
    string LockId   { get; }          // 唯一标识(用于安全释放)
    Task<bool> ReleaseAsync();
    Task<bool> RenewAsync(TimeSpan extend);   // 续约(看门狗)
}
```

**Redlock 算法实现要点**（`RedLockProvider`）：
1. 向 N 个独立 Redis 节点（本工程 6 节点 Cluster，Redlock 用其中 5 个 master）并发 `SET key value NX PX expiry`
2. 统计成功节点数 ≥ `ceil((N+1)/2)` 即多数派 → 加锁成功
3. 计算加锁耗时，若剩余有效期 ≤0 则视为失败，回滚已获锁节点
4. 释放时向所有节点发送 Lua 脚本：`if get==lockId then del`
5. **看门狗**：后台任务每 `expiry/3` 续约一次，防止业务未完成锁过期

**学习要点注释**：实现类中逐行注释 Redlock 的"为什么"——为什么需要多数派、为什么要校验耗时、为什么释放用 Lua 脚本保证原子性。

**调用示例**：
```csharp
await using var handle = await _lock.TryAcquireAsync(
    resource: $"payment:callback:{orderId}",
    expiryTime: TimeSpan.FromSeconds(10));

if (handle is null)
    return Result.Fail("订单处理中，请稍后");   // 重复回调被拦截

// ... 临界区：处理回调、记账
```

### 4.2 ZooKeeper 分布式锁

**设计目标**：资金账户变动前严格互斥，彻底杜绝并发更新脏读与资金多记。ZK 锁基于"临时顺序节点"，比 Redis 锁更强的一致性保证。

**实现要点**（`ZooKeeperLockProvider`）：
1. 在锁根节点 `/locks/account/{accountId}` 下创建临时顺序节点 `lock-00000001`
2. 获取所有子节点，判断自己是否最小节点
3. 是 → 获得锁；否 → 监听前一个节点删除事件（避免惊群）
4. 业务完成或会话断开 → 临时节点自动删除，释放锁
5. 设置 `sessionTimeout` 防止 GC 长暂停导致"假死"

**学习要点**：ZK 锁的 CP 特性（强一致）vs Redis 锁的 AP 特性（高可用）。代码注释对比两者适用场景——账户资金场景对一致性敏感，选 ZK；回调去重场景对延迟敏感，选 Redis。

### 4.3 ★ 双重锁方案 (DualLockProvider) - 核心

**设计思想**：Redis 锁快但极端情况下可能失效（如 GC 长暂停导致锁过期，业务仍在执行）；ZK 锁强一致但延迟较高。两者结合：

```
支付回调处理流程：
   ① Redis 锁拦截重复回调 (10ms 级，挡住 99% 重复)
      │
      ▼ 成功获取 Redis 锁
   ② 进入业务处理，记账前再获取 ZK 锁 (100ms 级，强一致)
      │
      ▼ 获得双重保护
   ③ 执行账户余额变更 + 流水写入 (DB 事务)
      │
      ▼ 提交事务后释放 ZK 锁
   ④ 发送 Kafka 支付成功事件，释放 Redis 锁
```

**接口设计**：
```csharp
public class DualLockProvider
{
    // Redis 快锁 + ZK 强一致锁组合
    public Task<ILockHandle?> AcquireForCallbackAsync(string orderId);
    public Task<ILockHandle?> AcquireForAccountAsync(long accountId);
}
```

**学习要点**：双重锁不是简单串联，而是"分层防御"——Redis 锁保护重复请求入口，ZK 锁保护资金变更临界区。注释中给出"如果只用一种锁会出什么问题"的反例分析。

### 4.4 Kafka 事件总线

**主题设计**：

| Topic | 用途 | 分区键 | 分区数 |
|-------|------|--------|-------|
| `payment-events-order-created` | 订单创建事件 | orderId | 6 |
| `payment-events-callback-received` | 回调到达事件 | orderId | 6 |
| `payment-events-account-changed` | 账户变更事件 | accountId | 12 |
| `payment-events-notify-merchant` | 商户通知任务 | merchantId | 6 |
| `payment-dlq` | 死信队列（消费失败转移） | - | 3 |

**生产者封装**（`KafkaEventBus`）：
- 基于 `Confluent.Kafka` Producer
- 同步发送 + acks=all（金融场景要求不丢消息）
- 内置 `EnableIdempotence=true` 幂等生产者
- 失败重试 + 指数退避（Polly）

**消费者封装**（`KafkaConsumer<T>`）：
- 抽象基类，子类只需实现 `HandleAsync(T evt)`
- 手动提交 Offset（确保处理成功后再提交）
- 死信队列流转：连续失败 N 次转入 `payment-dlq`
- 优雅关闭：`Consumer.Close()` 保证 Rebalance 快速完成

**单体架构下的 Kafka 用法说明**：虽然是单体应用，但故意使用 Kafka 处理异步事件（回调→记账→通知），目的是：
1. 学习 Kafka 生产/消费完整链路
2. 实现真正的异步解耦（回调快速返回，记账异步执行）
3. 为未来拆分微服务做准备（拆分后各服务消费各自 Topic）

**学习要点**：代码注释解释 Kafka 三种交付语义（At-most-once / At-least-once / Exactly-once），并说明本工程如何通过"幂等消费 + 事务"达到业务上的 Exactly-once。

### 4.5 ZeroMQ 通道通信

**场景**：Channels 模块内部模拟"路由→通道适配"的高频低延迟通信。单体架构下通过本地环回（127.0.0.1）学习 ZeroMQ 通信模式，未来拆分微服务后改为跨主机通信。

**两种模式封装**：
- `ZeroMqReqRepClient`：REQ/REP 同步请求-响应（查询通道状态、发起支付）
- `ZeroMqPubSubClient`：PUB/SUB 异步广播（通道状态变更通知）

**学习要点**：ZeroMQ 不是消息队列，而是"智能套接字库"。代码注释对比 ZeroMQ vs HTTP vs gRPC 的延迟、可靠性、使用复杂度。

### 4.6 Redis-Cluster 缓存 + 防穿透雪崩

**缓存设计**：
- 缓存支付渠道配置、商户信息、费率（读多写少）
- 缓存订单查询结果（短期，5-30秒）

**防穿透**（`CachePenetrationGuard`）：
- 布隆过滤器拦截不存在的 Key
- 空值缓存（短 TTL，如 60s）防止反复查 DB

**防雪崩**：
- 过期时间加随机抖动（`ttl + random(0, 60s)`）
- 互斥重建（缓存失效时只允许一个请求回源 DB，其他等待）

## 五、通用组件设计封装

### 5.1 异步处理框架

**目标**：将支付回调响应时间从 500ms+ 优化至 100ms 内。

**核心机制**：
1. **同步路径**：仅做幂等校验 + Redis 锁 + 状态机翻转（<50ms 立即返回 ACK）
2. **异步路径**：耗时业务（记账、通知商户）通过 Kafka 事件异步执行
3. **并行化**：使用 `Task.WhenAll` 并行执行无依赖任务

**Channel + 后台任务**（.NET 高性能队列）：
```csharp
// 内部进程内队列，承接 Kafka 消费后的二级处理
Channel<PaymentCallbackEvent> _channel = Channel.CreateBounded<PaymentCallbackEvent>(1000);

// 多消费者并行处理
for (int i = 0; i < Environment.ProcessorCount; i++)
{
    _ = Task.Run(ConsumerLoop);
}
```

**学习要点**：注释对比 `Task.Run`、`ThreadPool.QueueUserWorkItem`、`Channel` 三种方式，解释为什么高吞吐场景应使用 `Channel<T>`（背压控制、零分配）。

### 5.2 链路追踪 (OpenTelemetry + Jaeger)

**设计**：使用 .NET 官方 OpenTelemetry SDK，自动埋点 HTTP / SqlSugar / Kafka / Redis / ZeroMQ。

**配置**：
```csharp
services.AddOpenTelemetry()
    .WithTracing(tp => tp
        .AddAspNetCore()                  // HTTP 入口
        .AddSqlSugarInstrumentation()     // DB 调用 (SqlSugar 通过 ADO.NET 自动埋点)
        .AddSource("PaymentGateway")      // 自定义埋点
        .AddOtlpExporter(o =>             // 上报 Jaeger
            o.Endpoint = new Uri("http://jaeger:4317")));
```

**自定义埋点**：
```csharp
using var activity = _activitySource.StartActivity("ProcessCallback");
activity?.SetTag("orderId", orderId);
activity?.SetTag("channel", "wechat");
```

**学习要点**：Trace/Span 概念、采样率、Baggage 跨服务传递上下文、与日志关联（日志输出 traceId）。

### 5.3 统一异常处理 + 全局返回

**`ExceptionMiddleware`**：捕获所有未处理异常，按类型分类返回：
- `BusinessException` → 400 + 业务错误码
- `IdempotentException` → 409 + 重复请求提示
- 其他 → 500 + traceId（用于排查）

**统一返回结构**：
```csharp
public class Result<T>
{
    public bool   Success { get; init; }
    public string? Code    { get; init; }     // "OK" / "BIZ_001" 等
    public string? Message { get; init; }
    public T?     Data    { get; init; }
    public string? TraceId { get; init; }      // 便于排查
}
```

### 5.4 幂等控制

**两层幂等**：
1. **HTTP 入口幂等**：`[Idempotent]` 特性 + 请求头 `Idempotency-Key`，5 分钟内同 Key 返回首次结果
2. **业务幂等**：`payment_records` 表 `channel + channel_order_no` 唯一索引，DB 层兜底

**学习要点**：为什么金融场景需要"多层幂等"——网络重试、回调重发、用户重复点击，任何一层失效都不应导致重复入账。

### 5.5 限流熔断 (Polly)

- **限流**：基于 Redis 的滑动窗口算法，限制单商户 QPS
- **熔断**：HttpClientFactory + Polly，第三方通道异常率超阈值自动熔断降级
- **超时**：所有外部调用 5s 超时

### 5.6 配置 + 健康检查

- 配置强类型绑定 `IOptions<T>` + `Validate` 启动期校验
- 健康检查：PG 连接、Redis 连接、Kafka 连通性、ZK 会话状态
- `/health/live` vs `/health/ready` 区分探针

### 6.2 ★ 支付回调流程 (核心重点)

这是分布式锁、Kafka、异步处理三大技术点的综合体现场景。

```
微信/支付宝/银联回调
 │ POST /api/v1/callbacks/wechat
 ▼
┌─────────────────────────────────────────────────────────┐
│ CallbackEndpoints                                        │
│  ① 验签(防篡改)                                          │
│  ② 解析 channel_order_no + 状态                           │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ ★ Redis 锁 TryAcquire($"callback:{channel_order_no}")     │
│   失败 → 返回成功(渠道不重试)，记录"重复回调"               │
│   成功 → 进入处理                                          │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ 同步快速路径 (50ms 内返回 ACK)                            │
│  ③ 校验订单存在性 + 状态机翻转(待支付→支付中→已支付)        │
│  ④ 更新 payment_records 状态                             │
│  ⑤ 发布 CallbackReceivedEvent 到 Kafka                   │
│  ⑥ 返回 ACK 给渠道                                       │
└──────────────────────┬──────────────────────────────────┘
                       ▼ (异步 Kafka 消费)
┌─────────────────────────────────────────────────────────┐
│ Accounts 模块 - CreditAccountConsumer (Kafka 消费者)      │
│  ⑦ ★ ZK 锁 Acquire($"account:{accountId}")               │
│  ⑧ 开启 DB 事务                                          │
│  ⑨ SELECT FOR UPDATE 锁定账户行(乐观锁兜底)               │
│  ⑩ 余额变更 + 写入流水 account_transactions               │
│  ⑪ 提交事务 → 释放 ZK 锁                                  │
│  ⑫ 发布 AccountChangedEvent                              │
└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
│ Notify 模块 - MerchantNotifyConsumer (Kafka 消费者)       │
│  ⑬ 异步通知商户支付结果(HTTP 回调 + 重试)                  │
│  ⑭ 失败转入死信队列                                       │
└─────────────────────────────────────────────────────────┘
```

**性能优化点（学习要点）**：
- 同步路径只做"状态翻转 + 事件发布"，不阻塞在记账上 → 100ms 内响应
- 记账走 Kafka 异步 → 不影响回调响应时间
- Kafka 消费者并行处理 → 吞吐量提升
- DB 事务粒度最小化（仅锁账户行）

### 8.2 应用 Dockerfile (多阶段构建)

```dockerfile
# 学习要点：多阶段构建(减小最终镜像体积)
# 阶段1: 构建(用SDK镜像)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["src/PaymentGateway.Api/PaymentGateway.Api.csproj", "PaymentGateway.Api/"]
RUN dotnet restore "PaymentGateway.Api/PaymentGateway.Api.csproj"
COPY . .
WORKDIR "/src/PaymentGateway.Api"
RUN dotnet publish -c Release -o /app/publish --no-restore

# 阶段2: 运行(用runtime镜像,体积小)
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
WORKDIR /app
# aspnet:8.0 基础镜像不含 curl,需安装供 HEALTHCHECK 使用
RUN apt-get update && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*
COPY --from=build /app/publish .
EXPOSE 5000
ENV ASPNETCORE_URLS=http://+:5000
# ★ M5-3: 设置 ASPNETCORE_ENVIRONMENT=Docker,触发加载 appsettings.Docker.json
#   (容器内用服务名 redis-1/zookeeper/kafka 互访,而非 localhost)
ENV ASPNETCORE_ENVIRONMENT=Docker
HEALTHCHECK --interval=30s --timeout=3s \
    CMD curl -f http://localhost:5000/health || exit 1
ENTRYPOINT ["dotnet", "PaymentGateway.Api.dll"]
```

**为什么多阶段**：SDK 镜像 ~1GB，runtime 镜像 ~200MB。构建用 SDK，运行用 runtime，最终镜像体积小、攻击面小。

**★ M5-3 容器化配置覆盖**：容器内应用通过 `appsettings.Docker.json` 覆盖 `appsettings.json` 的连接配置:
- MySQL: `Server=host.docker.internal` (容器访问宿主机 MySQL)
- Redis/ZK/Kafka/Jaeger: 用容器间服务名 (`redis-1:7000` / `zookeeper:2181` / `kafka:9092` / `jaeger:4317`)
- `EventBus:UseInMemory=false` (切换为真实 Kafka 事件驱动联调)
