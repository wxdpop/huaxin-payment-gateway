## 1. .NET 8 与 ASP.NET Core 底层特性

### 1.1 GC 分代回收

**原理**：.NET GC 基于"弱代假设"——新对象朝生夕死，老对象倾向于存活。分 3 代回收以减少全堆扫描成本。

```
┌──────────┐     幸存晋升     ┌──────────┐     幸存晋升     ┌──────────────┐
│  Gen 0   │  ──────────►    │  Gen 1   │  ──────────►    │   Gen 2      │
│ (小容量)  │                 │ (中容量)  │                 │ (大对象LOH)   │
│ 回收最频繁│                 │ 回收次之  │                 │ 回收最昂贵    │
└──────────┘                 └──────────┘                 └──────────────┘
   Ephemeral (代际)                                          Full GC (STW)
```

**回收触发**：Gen0 满触发 Gen0 GC；Gen1 满触发 Gen1+Gen0；Gen2 满触发 Full GC（STW 暂停全部线程）。

**支付项目中的注意点**：
- 高并发回调每秒数千次对象分配，若全部进 Gen2 会引发频繁 Full GC → 卡顿
- 解决：使用 `ArrayPool<T>`、`Channel<T>`、对象池减少分配；大缓冲区用 `ArrayPool<byte>.Shared.Rent()`

```csharp
// 学习要点：减少 Gen2 压力的对象池用法
public class CallbackBufferPool
{
    private readonly ConcurrentBag<byte[]> _pool = new();

    public byte[] Rent() => _pool.TryTake(out var buf) ? buf : new byte[4096];

    public void Return(byte[] buf) => _pool.Add(buf);  // 复用，避免反复分配进Gen2
}

// 高并发读取回调报文的正确姿势
public async Task<byte[]> ReadCallbackAsync(Stream stream, CancellationToken ct)
{
    byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);  // 租借，不new
    try
    {
        using var ms = new MemoryStream();
        int read;
        while ((read = await stream.ReadAsync(buffer.AsMemory(0, 4096), ct)) > 0)
            ms.Write(buffer, 0, read);
        return ms.ToArray();
    }
    finally
    {
        ArrayPool<byte>.Shared.Return(buffer);  // 必须归还
    }
}
```

**Server GC vs Workstation GC**：生产用 ServerGC（多CPU并行回收，吞吐高）。`runtimeconfig.json` 配置：
```json
{ "configProperties": { "System.Runtime.GC.ServerGC": true, "System.Runtime.GC.Concurrent": true } }
```

### 1.2 DI 生命周期

三种生命周期对比：

| 生命周期 | 接口 | 行为 | 支付项目场景 |
|---------|------|------|-------------|
| Transient | `ITransient` | 每次请求新建 | 轻量无状态服务（如金额计算器） |
| Scoped | `IScoped` | 每个 Scope 共享（HTTP请求=1Scope） | ISqlSugarClient、当前用户上下文 |
| Singleton | `ISingleton` | 全局唯一 | 配置、连接池、Kafka Producer |

**★ 陷阱：Captive Dependency（被囚禁依赖）**

```csharp
// ❌ 危险：Singleton 持有 Scoped 依赖
services.AddSingleton<IOrderCache, OrderCache>();
// 若 OrderCache 构造注入了 ISqlSugarClient → 客户端被长期持有
// → 跨请求复用 → 并发问题 + 跟踪错乱

// ✅ 正确：Singleton 只能注入 Singleton/Transient
public class OrderCache : IOrderCache
{
    private readonly IServiceProvider _sp;  // 注入 IServiceProvider
    public Order OrderById(long id)
    {
        using var scope = _sp.CreateScope();           // 按需创建 Scope
        var repo = scope.ServiceProvider
                        .GetRequiredService<IOrderRepo>();
        return repo.Get(id);
    }
}
```

**学习要点**：DI 容器解析时构造函数注入会"捕获"依赖，Singleton 抓住 Scoped 会让 Scoped 退化为 Singleton，是隐蔽 bug 来源。

### 1.3 中间件管道模型

ASP.NET Core 请求通过"管道"依次流经中间件，类似洋葱模型：

```
请求 →  [Exception]  →  [TraceId]  →  [Auth]  →  [RateLimit]  →  [Endpoint]
                                                                        ↓
响应 ←  [Exception]  ←  [TraceId]  ←  [Auth]  ←  [RateLimit]  ←  [Handler]
```

```csharp
// 学习要点：自定义中间件约定类写法
public class TraceIdMiddleware
{
    private readonly RequestDelegate _next;
    public TraceIdMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext ctx)
    {
        var traceId = ctx.Request.Headers["X-Trace-Id"].FirstOrDefault()
                   ?? Guid.NewGuid().ToString("N");
        ctx.Items["TraceId"] = traceId;
        ctx.Response.Headers["X-Trace-Id"] = traceId;

        Activity.Current?.SetTag("trace.id", traceId);  // 关联 Jaeger
        await _next(ctx);  // 继续管道
    }
}

// 注册（顺序很重要！异常中间件必须最外层）
app.UseMiddleware<ExceptionMiddleware>();   // 1. 最外层兜底异常
app.UseMiddleware<TraceIdMiddleware>();     // 2. 注入 TraceId
app.UseAuthentication();                    // 3. 鉴权
app.UseMiddleware<RateLimitMiddleware>();   // 4. 限流
app.UseAuthorization();                     // 5. 授权
app.MapControllers();                       // 6. 端点路由
```

**短路 vs 透传**：中间件可选择不调用 `_next` 实现短路（如限流直接返回 429）。管道顺序决定执行顺序——异常处理必须最先注册才能捕获后续所有异常。

### 1.4 async/await 异步编程

**状态机原理**：`async` 方法被编译器转换为状态机，`await` 处插入挂起点。线程不被阻塞，可服务于其他请求 → 提高吞吐。

```csharp
// 学习要点：回调处理同步 vs 异步路径对比
public class PaymentCallbackHandler
{
    // ❌ 旧实现：500ms+ 同步阻塞，占用线程
    public Result OldHandle(Callback cb)
    {
        var order = _orderRepo.FindByChannelOrderNo(cb.ChannelOrderNo);  // DB IO
        var account = _accountRepo.FindByMerchantId(order.MerchantId);   // DB IO
        account.Balance += order.Amount;
        _accountRepo.Update(account);
        _uow.Save();
        NotifyMerchant(order);  // HTTP 同步调用
        return Result.Ok();     // 总耗时 > 500ms
    }

    // ✅ 优化：100ms 内返回，异步处理耗时操作
    public async Task<Result> NewHandleAsync(Callback cb, CancellationToken ct)
    {
        // 同步快速路径：状态机翻转 + 事件发布
        var order = await _orderRepo.FindByChannelOrderNoAsync(cb.ChannelOrderNo, ct);
        order.MarkAsPaid();  // 领域逻辑
        await _orderRepo.UpdateAsync(order, ct);

        // ★ 发布事件而非直接记账（异步路径）
        await _eventBus.PublishAsync(new CallbackReceivedEvent(order.Id), ct);
        // 同步路径到此结束，<100ms 返回 ACK

        // 记账、通知商户由 Kafka 消费者异步执行
        return Result.Ok();
    }
}
```

**Task.WhenAll 并行化无依赖任务**：

```csharp
// 并行查询订单 + 账户 + 渠道（原本串行 300ms，并行后 100ms）
var orderTask   = _orderRepo.GetAsync(orderId, ct);
var accountTask = _accountRepo.GetAsync(accountId, ct);
var channelTask = _channelService.GetAsync(channelCode, ct);
await Task.WhenAll(orderTask, accountTask, channelTask);
// 三个任务并行，总耗时 = 最慢的那个
```

**★ 避坑：async void**：仅用于事件处理器，其他场景用 `async Task`。`async void` 异常无法被捕获，会拖垮进程。

**★ ConfigureAwait(false)**：库代码中应使用 `await foo.ConfigureAwait(false)` 避免捕获同步上下文，防止死锁。ASP.NET Core 本身无同步上下文，但养成习惯利于跨平台库。

## 2. PostgreSQL 数据库设计与优化

### 2.1 PostgreSQL vs MySQL 选型

| 维度 | PostgreSQL | MySQL |
|------|------------|-------|
| 事务 | MVCC（无锁读） | InnoDB MVCC |
| JSON | JSONB（二进制索引） | JSON（无索引优化） |
| 并发写 | 优（行级锁） | 优 |
| 金融场景 | ★ 推荐（强一致 + NUMERIC） | 可用 |
| 部署 | 主从 + 流复制 | 主从 + Group Replication |

**支付项目选 PG 原因**：`NUMERIC(18,2)` 精度无损失（浮点不能存金额！）；`SELECT FOR UPDATE` 行锁支持好；JSONB 存储费率配置灵活。

### 2.2 表设计与索引优化

```sql
-- ★ 金额字段必须 NUMERIC，绝不用 FLOAT/DOUBLE（精度丢失会导致资金差错）
amount NUMERIC(18,2) NOT NULL

-- 复合索引：按状态分页查询订单（高频场景）
CREATE INDEX idx_orders_status_created ON orders(status, created_at DESC);
-- 查询 WHERE status=0 ORDER BY created_at DESC LIMIT 20 可走索引覆盖

-- 部分索引：只为有效数据建索引（节省空间）
CREATE INDEX idx_orders_channel_no ON orders(channel_order_no)
    WHERE channel_order_no IS NOT NULL;

-- 唯一约束 = 业务幂等的最后一道防线
CONSTRAINT uk_channel_order UNIQUE (channel_code, channel_order_no)
```

**索引使用建议**：
- WHERE + ORDER BY 字段组合建复合索引（顺序：等值在前，范围在后）
- 高频更新的列少建索引（索引维护成本）
- `EXPLAIN ANALYZE` 验证执行计划，避免全表扫描

### 2.3 并发控制：乐观锁 vs 悲观锁

**悲观锁**（`SELECT FOR UPDATE`）：
```csharp
// 账户余额变更 - 悲观锁
public async Task<bool> DebitAsync(long accountId, decimal amount)
{
    await using var tx = await _db.Database.BeginTransactionAsync();
    // ★ SELECT FOR UPDATE 锁行，其他事务等待
    var account = await _db.Accounts
        .FromSqlRaw("SELECT * FROM accounts WHERE id={0} FOR UPDATE", accountId)
        .FirstAsync();
    if (account.Balance < amount) { await tx.RollbackAsync(); return false; }
    account.Balance -= amount;
    await _db.SaveChangesAsync();
    await tx.CommitAsync();
    return true;
}
```

**乐观锁**（version 字段）：
```csharp
// 账户变更 - 乐观锁（高并发读多写少场景）
public async Task<bool> UpdateBalanceAsync(long accountId, decimal delta, long oldVersion, CancellationToken ct)
{
    // UPDATE ... SET balance=balance+@delta, version=version+1
    // WHERE id=@id AND version=@oldVersion
    int affected = await _db.Accounts
        .Where(a => a.Id == accountId && a.Version == oldVersion)
        .ExecuteUpdateAsync(s => s
            .SetProperty(a => a.Balance, a => a.Balance + delta)
            .SetProperty(a => a.Version, a => a.Version + 1), ct);
    return affected > 0;  // 0表示版本冲突，需重试
}
```

**支付项目选型**：账户余额变更用"ZK锁 + 乐观锁兜底"双重保险；订单状态用乐观锁（version）。

### 2.4 事务隔离级别

PostgreSQL 默认 Read Committed。支付场景账户变更需要 Serializable 或显式行锁：

## 3. Redis-Cluster 分布式锁 (Redlock)

### 3.1 单节点 Redis 锁的局限

```csharp
// 简单 SETNX 锁 - 单节点
public bool Lock(string key, string value, TimeSpan expiry)
    => _redis.StringSet(key, value, expiry, When.NotExists);  // SET NX PX
```

**问题**：单点故障时锁失效；主从切换瞬间可能丢失锁（主写入但未同步到从，主宕机，从提升为新主，锁丢失）。

### 3.2 Redlock 算法

**核心思想**：向 N 个独立 Redis 节点（奇数，推荐 5）申请锁，多数派（≥3）成功才算加锁成功。任何少数节点故障不影响锁正确性。

**算法步骤**：
1. 获取当前毫秒时间 `T1`
2. 向所有 N 个节点并发 `SET key value NX PX expiry`
3. 统计成功节点数 `quorum`
4. 计算耗时 `cost = T2 - T1`
5. 若 `quorum >= (N/2 + 1)` 且 `expiry - cost > 0` → 加锁成功
6. 否则 → 向所有节点发送释放（即使没加锁的也尝试释放）

```csharp
// 学习要点：Redlock 算法实现
public class RedLockProvider : IDistributedLock
{
    private readonly List<IConnectionMultiplexer> _redises;  // N个独立Redis实例
    private readonly int _quorum;                            // 多数派阈值

    public RedLockProvider(IEnumerable<IConnectionMultiplexer> redises)
    {
        _redises = redises.ToList();
        _quorum = _redises.Count / 2 + 1;  // N=5 → quorum=3
    }

    public async Task<ILockHandle?> TryAcquireAsync(
        string resource, TimeSpan expiry, CancellationToken ct = default)
    {
        var lockId = Guid.NewGuid().ToString("N");
        var ttlMs   = (int)expiry.TotalMilliseconds;

        // ★ 1. 并发向所有节点申请锁
        var tasks = _redises.Select(r => TryLockOneAsync(r, resource, lockId, ttlMs, ct));
        var results = await Task.WhenAll(tasks);

        // ★ 2. 计算耗时（扣除时钟漂移）
        var costMs = (DateTime.UtcNow - DateTimeOffset.UtcNow).TotalMilliseconds; // 简化
        // 实际应记录开始时间 T1
        var driftMs = ttlMs * 0.01 + 2;  // 时钟漂移补偿
        var validityTime = ttlMs - costMs - driftMs;

        // ★ 3. 多数派校验 + 有效期校验
        var successCount = results.Count(r => r);
        if (successCount >= _quorum && validityTime > 0)
        {
            return new RedLockHandle(this, resource, lockId, TimeSpan.FromMilliseconds(validityTime));
        }

        // ★ 4. 失败：必须释放已获锁的节点（防止脏锁残留）
        await ReleaseInternalAsync(resource, lockId);
        return null;
    }

    private static async Task<bool> TryLockOneAsync(
        IConnectionMultiplexer redis, string resource, string lockId, int ttlMs, CancellationToken ct)
    {
        try
        {
            var db = redis.GetDatabase();
            // SET key value NX PX ttl  - 原子操作
            return await db.StringSetAsync(resource, lockId, TimeSpan.FromMilliseconds(ttlMs),
                When.NotExists, CommandFlags.DemandMaster);
        }
        catch { return false; }  // 单节点异常不影响整体
    }

    // ★ 释放锁必须用 Lua 脚本保证原子性（GET+DEL不能拆开）
    private const string UnlockScript = @"
        if redis.call('get', KEYS[1]) == ARGV[1] then
            return redis.call('del', KEYS[1])
        else
            return 0
        end";

    public async Task<bool> ReleaseAsync(string resource, string lockId)
    {
        var tasks = _redises.Select(r =>
        {
            var db = r.GetDatabase();
            return (Task<bool>)db.ScriptEvaluateAsync(UnlockScript,
                new RedisKey[] { resource }, new RedisValue[] { lockId });
        });
        var results = await Task.WhenAll(tasks);
        return results.Count(x => x) >= _quorum;
    }
}
```

**★ 为什么释放用 Lua 脚本**：`GET` 然后判断再 `DEL` 不是原子的，可能 A 进程判断是自己的锁，B 进程的锁（已过期后重新分配给B）正好此时被 A 误删。Lua 脚本在 Redis 单线程内原子执行。

### 3.3 看门狗（Watchdog）续约

防止业务执行时间超过锁 TTL 导致锁过期：

```csharp
public class RedLockHandle : ILockHandle
{
    private readonly Timer _watchdog;  // 看门狗定时器

    public RedLockHandle(RedLockProvider provider, string resource, string lockId, TimeSpan validity)
    {
        _provider = provider;
        Resource = resource;
        LockId   = lockId;

        // 每 validity/3 续约一次
        var renewInterval = TimeSpan.FromMilliseconds(validity.TotalMilliseconds / 3);
        _watchdog = new Timer(async _ => await RenewAsync(renewInterval * 2), null,
            renewInterval, renewInterval);
    }

    public async Task<bool> RenewAsync(TimeSpan extend)
    {
        // Lua: if get==lockId then pexpire
        return await _provider.RenewInternalAsync(Resource, LockId, extend);
    }

    public async ValueTask DisposeAsync()
    {
        await _watchdog.DisposeAsync();
        await _provider.ReleaseAsync(Resource, LockId);
    }
}
```

**支付项目应用**：回调处理以 `callback:{channelOrderNo}` 为 Key 获取 Redis 锁，10s TTL，挡住重复回调。学习要点：Redlock 的"多数派"思想与 Paxos/Raft 一脉相承，理解它有助于学习一致性算法。

## 4. ZooKeeper 分布式锁

### 4.1 ZK 数据模型与锁原理

ZK 是强一致（CP）的分布式协调服务，数据模型类似文件系统的"树形 znode"。基于"临时顺序节点"实现公平锁：

**算法步骤**：
1. 在锁根节点 `/locks/account/{accountId}` 下创建临时顺序节点 `lock-00000001`
2. 获取所有子节点列表
3. 判断自己是否最小节点
4. 是 → 获得锁；否 → 监听前一个节点删除事件（避免惊群）
5. 业务完成或会话断开 → 临时节点自动删除，后继者被唤醒

```csharp
// 学习要点：ZK 分布式锁实现
public class ZooKeeperLockProvider : IDistributedLock
{
    private readonly IZooKeeper _zk;
    private readonly TimeSpan _sessionTimeout;

    public async Task<ILockHandle?> TryAcquireAsync(
        string resource, TimeSpan expiry, CancellationToken ct = default)
    {
        var lockPath = $"/locks/{resource.Replace(':', '/')}";
        await EnsurePathAsync(lockPath);  // 创建持久父节点

        // ★ 1. 创建临时顺序节点 (EPHEMERAL_SEQUENTIAL)
        var myNode = await _zk.CreateAsync(
            $"{lockPath}/lock-",                     // 节点前缀
            data: Encoding.UTF8.GetBytes(_clientId),
            acls: ZooDefs.Ids.OPEN_ACL_UNSAFE,
            CreateMode.EPHEMERAL_SEQUENTIAL);        // ★ 临时+顺序
        // 返回如 /locks/account/1001/lock-00000007

        // ★ 2. 获取所有子节点并排序
        var children = await _zk.GetChildrenAsync(lockPath);
        var sorted = children.OrderBy(n => n).ToList();
        var mySeq = myNode.Substring(myNode.LastIndexOf('/') + 1);

        // ★ 3. 判断是否最小节点
        if (sorted[0] == mySeq)
            return new ZkLockHandle(_zk, myNode);   // 获得锁

        // ★ 4. 不是最小 → 监听前一个节点（避免惊群，只监听前驱）
        var myIndex = sorted.IndexOf(mySeq);
        var prevNode = sorted[myIndex - 1];
        var waitNode = $"{lockPath}/{prevNode}";

        var tcs = new TaskCompletionSource<bool>();
        using var reg = _zk.Register(WatcherType.Children, waitNode, new Watcher
        {
            Process = e =>
            {
                if (e.Type == EventType.NodeDeleted)
                    tcs.TrySetResult(true);  // 前驱删除，轮到自己
                return Task.CompletedTask;
            }
        });

        // 检查前驱是否已删除（避免错过事件）
        if (!await _zk.ExistsAsync(waitNode))
            return new ZkLockHandle(_zk, myNode);

        // 等待前驱删除或超时
        using var cts = CancellationTokenSource.CreateLinkedTokenSource(ct);
        cts.CancelAfter(expiry);
        using cts.Token.Register(() => tcs.TrySetCanceled());
        await tcs.Task;  // 阻塞等待
        return new ZkLockHandle(_zk, myNode);
    }
}
```

### 4.2 ZK 锁 vs Redis 锁

| 维度 | ZooKeeper 锁 | Redis 锁 (Redlock) |
|------|--------------|-------------------|
| 一致性 | CP（强一致） | AP（高可用） |
| 延迟 | 较高（100ms+） | 低（10ms级） |
| 可靠性 | 极高（临时节点+会话） | 依赖 TTL + 看门狗 |
| 复杂度 | 高（需维护会话/监听） | 中 |
| 适用场景 | 资金账户变更（强一致） | 回调去重（低延迟） |

**学习要点**：ZK 临时节点与会话绑定，客户端断连自动删除，天然防止"持锁者宕机导致死锁"。但 GC 长暂停可能导致会话超时被误判宕机，需设置合理 `sessionTimeout`。

## 5. 双重锁方案 (Redis + ZK)

### 5.1 为什么需要双重锁

**单 Redis 锁的风险**：极端情况下 GC 长暂停（>TTL）或网络分区，Redis 锁可能过期，但业务仍在执行 → 此时另一请求获取到锁 → 重复入账。

**单 ZK 锁的不足**：每次加锁需创建节点+监听，延迟高（100ms+），不适合高频回调去重。

**双重锁分层防御**：Redis 锁挡住 99% 重复请求（快），ZK 锁保护资金变更临界区（稳）。

```
重复回调请求
     │
     ▼
★ Redis 锁 (callback:{orderId})  ── 失败 → 拦截 (10ms级)
     │ 成功
     ▼
   处理订单状态机
     │
     ▼  (记账前)
★ ZK 锁 (account:{accountId})     ── 失败 → 等待/重试 (100ms级)
     │ 成功
     ▼
   DB 事务: 余额变更 + 流水
     │ 提交
     ▼
   释放 ZK 锁
     │
     ▼
   发布 Kafka 事件
     │
     ▼
   释放 Redis 锁
```

```csharp
// 学习要点：双重锁封装
public class DualLockProvider
{
    private readonly IDistributedLock _redisLock;  // Redlock
    private readonly IDistributedLock _zkLock;      // ZooKeeper

    /// <summary>回调处理双重锁（Redis快锁为主）</summary>
    public async Task<ILockHandle?> AcquireForCallbackAsync(string orderId)
    {
        // Redis 锁即可（回调场景对延迟敏感，可容忍极小概率失效，有ZK兜底）
        return await _redisLock.TryAcquireAsync(
            $"payment:callback:{orderId}", TimeSpan.FromSeconds(10));
    }

    /// <summary>资金账户变更双重锁（Redis + ZK）</summary>
    public async Task<AccountLockHandle?> AcquireForAccountAsync(long accountId)
    {
        // 先 Redis 快锁（挡住并发请求）
        var redisHandle = await _redisLock.TryAcquireAsync(
            $"payment:account:{accountId}", TimeSpan.FromSeconds(15));
        if (redisHandle is null) return null;

        try
        {
            // 再 ZK 强一致锁（防 GC 暂停等极端场景）
            var zkHandle = await _zkLock.TryAcquireAsync(
                $"account:{accountId}", TimeSpan.FromSeconds(15));
            if (zkHandle is null)
            {
                await redisHandle.DisposeAsync();  // 回滚 Redis 锁
                return null;
            }
            return new AccountLockHandle(redisHandle, zkHandle);
        }
        catch
        {
            await redisHandle.DisposeAsync();
            throw;
        }
    }
}

public sealed class AccountLockHandle : ILockHandle
{
    private readonly ILockHandle _redis;
    private readonly ILockHandle _zk;

    public async ValueTask DisposeAsync()
    {
        // ★ 释放顺序：先 ZK 再 Redis（与获取相反）
        await _zk.DisposeAsync();
        await _redis.DisposeAsync();
    }
}
```

**学习要点**：双重锁的"释放顺序"很关键——先释放 ZK（让后继者能获取），再释放 Redis（让外层请求能进入）。代码注释中详细分析"如果只用一种锁会出什么问题"的反例。

### 6.3 消费者封装

```csharp
// 学习要点：Kafka 消费者基类(手动Offset提交 + 死信队列)
public abstract class KafkaConsumer<TEvent> : BackgroundService where TEvent : IEvent
{
    private readonly IConsumer<string, string> _consumer;
    private readonly IEventBus _eventBus;       // 用于发送死信
    private readonly ILogger _logger;
    private readonly int _maxRetry = 3;

    protected KafkaConsumer(IConfiguration config, IEventBus eventBus, ILogger logger)
    {
        var cfg = new ConsumerConfig
        {
            BootstrapServers = config["Kafka:BootstrapServers"],
            GroupId = typeof(TEvent).Name + "-group",     // 消费者组
            EnableAutoCommit = false,                      // ★ 手动提交(防丢)
            AutoOffsetReset = AutoOffsetReset.Earliest,
            MaxPollIntervalMs = 300000                     // 长处理防Rebalance
        };
        _consumer = new ConsumerBuilder<string, string>(cfg).Build();
        _eventBus = eventBus;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        // ★ 订阅主题,循环消费
        _consumer.Subscribe(typeof(TEvent).Name);

        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var cr = _consumer.Consume(stoppingToken);  // 阻塞拉取
                var evt = JsonSerializer.Deserialize<TEvent>(cr.Message.Value);
                if (evt is null) continue;

                // ★ 幂等检查(消费侧也需幂等,因为at-least-once)
                if (await IsProcessedAsync(evt.EventId))
                {
                    _consumer.Commit(cr);   // 已处理,直接提交
                    continue;
                }

                await HandleWithRetryAsync(evt, stoppingToken);
                _consumer.Commit(cr);       // ★ 处理成功才提交Offset
            }
            catch (ConsumeException ex)
            {
                _logger.LogError(ex, "消费异常");
            }
            catch (OperationCanceledException) { break; }
        }

        _consumer.Close();   // ★ 优雅关闭(触发Rebalance,让分区快速reassign)
    }

    private async Task HandleWithRetryAsync(TEvent evt, CancellationToken ct)
    {
        int retry = 0;
        while (true)
        {
            try
            {
                await HandleAsync(evt, ct);
                return;
            }
            catch (Exception ex)
            {
                retry++;
                if (retry >= _maxRetry)
                {
                    // ★ 死信队列:超过重试次数,转入DLQ待人工处理
                    await _eventBus.PublishAsync(new DeadLetterEvent(evt, ex.Message), ct);
                    _logger.LogError(ex, "转入死信队列 eventId={Id}", evt.EventId);
                    return;
                }
                await Task.Delay(TimeSpan.FromSeconds(Math.Pow(2, retry)), ct);  // 指数退避
            }
        }
    }

    protected abstract Task HandleAsync(TEvent evt, CancellationToken ct);
    protected abstract Task<bool> IsProcessedAsync(string eventId);
}
```

### 6.4 交付语义

| 语义 | 实现 | 支付项目 |
|------|------|---------|
| At-most-once | 自动提交 + 先提交后处理 | ❌ 会丢消息 |
| At-least-once | 手动提交 + 先处理后提交 | ★ 本项目(配合幂等消费) |
| Exactly-once | 事务生产者 + 事务消费 | 复杂，业务层用幂等替代 |

**学习要点**：Kafka 的 Exactly-once 代价高，金融项目常用"at-least-once + 业务幂等"达到"业务层 exactly-once"。

## 8. Redis 缓存：雪崩/穿透/击穿

### 8.1 三大问题对比

| 问题 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 缓存雪崩 | 大量Key同时失效，请求全压DB | 批量Key相同TTL | TTL加随机 |
| 缓存穿透 | 查询不存在数据，缓存永不命中 | 恶意攻击/查询不存在 | 布隆过滤器+空值缓存 |
| 缓存击穿 | 热点Key失效瞬间，大量请求打DB | 热点Key过期 | 互斥重建+逻辑过期 |

### 8.2 防穿透实现

```csharp
// 学习要点：布隆过滤器 + 空值缓存防穿透
public class CachePenetrationGuard
{
    private readonly IConnectionMultiplexer _redis;
    private readonly BloomFilter _bloom;   // 布隆过滤器(可选:如BloomFilter.Net)

    public async Task<T?> GetAsync<T>(string key, Func<Task<T?>> loader)
    {
        // 1. 布隆过滤器拦截(数据存在性预判)
        if (!_bloom.Contains(key))
            return default;  // 一定不存在,直接返回

        // 2. 查缓存
        var cached = await _redis.GetDatabase().StringGetAsync(key);
        if (cached.HasValue) return Deserialize<T>(cached);

        // 3. 查DB
        var data = await loader();
        if (data is null)
        {
            // ★ 空值缓存(短TTL),防止反复查DB
            await _redis.GetDatabase().StringSetAsync(key, "NULL",
                TimeSpan.FromSeconds(60));
            return default;
        }

        await _redis.GetDatabase().StringSetAsync(key, Serialize(data),
            TimeSpan.FromMinutes(5) + RandomJitter());  // ★ 加随机抖动防雪崩
        return data;
    }

    private static TimeSpan RandomJitter() =>
        TimeSpan.FromSeconds(Random.Shared.Next(0, 60));
}
```

### 8.3 防击穿：互斥重建

```csharp
// 学习要点：缓存击穿互斥重建(单个请求回源,其他等待)
private readonly SemaphoreSlim _gate = new(1, 1);

public async Task<T?> GetWithMutexAsync<T>(string key, Func<Task<T?>> loader)
{
    var data = await _cache.GetAsync<T>(key);
    if (data is not null) return data;

    // ★ 抢锁重建(只允许一个请求查DB)
    await _gate.WaitAsync();
    try
    {
        // double-check(可能其他请求已重建)
        data = await _cache.GetAsync<T>(key);
        if (data is not null) return data;

        data = await loader();
        await _cache.SetAsync(key, data, TtlWithJitter());
        return data;
    }
    finally { _gate.Release(); }
}
```

## 11. 高并发资金安全保障

### 11.1 多层幂等防护

```
HTTP 层:    [Idempotent] 特性 + Idempotency-Key (5分钟)
业务层:     Redis 锁 (callback:{orderId})
DB 层:      payment_records 唯一约束 (channel + channel_order_no)
账户层:     account_transactions 唯一约束 (biz_no)
```

```csharp
// 学习要点：幂等特性(基于Idempotency-Key)
[AttributeUsage(AttributeTargets.Method)]
public class IdempotentAttribute : Attribute, IAsyncActionFilter
{
    public async Task OnActionExecutionAsync(ActionExecutingContext ctx, ActionExecutionDelegate next)
    {
        var key = ctx.HttpContext.Request.Headers["Idempotency-Key"].FirstOrDefault();
        if (string.IsNullOrEmpty(key))
        {
            ctx.Result = new BadRequestObjectResult("缺少 Idempotency-Key");
            return;
        }

        var svc = ctx.HttpContext.RequestServices.GetRequiredService<IIdempotentService>();
        // ★ SETNX 占位(5分钟)
        var acquired = await svc.TryAcquireAsync(key, TimeSpan.FromMinutes(5));
        if (!acquired)
        {
            // 返回首次响应(缓存)
            var cached = await svc.GetResponseAsync(key);
            ctx.Result = new ContentResult
            {
                Content = cached.ResponseBody,
                StatusCode = cached.StatusCode
            };
            return;
        }

        var executed = await next();   // 执行Action
        // 缓存响应供重试返回
        if (executed.Result is ObjectResult obj)
            await svc.CacheResponseAsync(key, obj.Value?.ToString() ?? "", 200);
    }
}
```

## 12. Docker 与 CI/CD

### 12.1 多阶段 Dockerfile

```dockerfile
# 学习要点：多阶段构建(减小最终镜像体积)
# 阶段1: 构建(用SDK镜像)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["src/PaymentGateway.Api/PaymentGateway.Api.csproj", "PaymentGateway.Api/"]
RUN dotnet restore "PaymentGateway.Api/PaymentGateway.Api.csproj"
COPY . .
WORKDIR "/src/PaymentGateway.Api"
RUN dotnet publish -c Release -o /app/publish --no-restore

# 阶段2: 运行(用runtime镜像,体积小)
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS final
WORKDIR /app
COPY --from=build /app/publish .
EXPOSE 5000
ENV ASPNETCORE_URLS=http://+:5000
HEALTHCHECK --interval=30s --timeout=3s \
    CMD curl -f http://localhost:5000/health/live || exit 1
ENTRYPOINT ["dotnet", "PaymentGateway.Api.dll"]
```

**为什么多阶段**：SDK 镜像 ~1GB，runtime 镜像 ~200MB。构建用 SDK，运行用 runtime，最终镜像体积小、攻击面小。
