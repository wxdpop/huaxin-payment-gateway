# MT5 平台详解与实际对接场景

> 面向 .NET 工程师的 MT5 深度学习文档，覆盖平台架构、业务场景、对接方式、代码示例。

## 1. MT5 是什么

**MetaTrader 5（MT5）** 是俄罗斯 MetaQuotes 公司于 2010 年推出的**多资产零售交易平台**，MT4 的下一代版本。

### 1.1 平台定位

| 维度 | 说明 |
|------|------|
| **业务领域** | 外汇、股票、期货、CFD、加密货币 |
| **市场地位** | 全球零售外汇/CFD 经纪商首选（市占率 > 70%） |
| **核心能力** | 行情接入、订单管理、图表分析、自动化交易（EA）、策略回测 |
| **编程语言** | MQL5（C++ 风格，面向对象增强） |
| **协议** | 私有协议（TCP），不公开 |

### 1.2 MT5 vs MT4 深度对比

| 维度 | MT4 | MT5 |
|------|-----|-----|
| **发布年份** | 2005 | 2010 |
| **资产支持** | 仅外汇 + CFD | 外汇 + 股票 + 期货 + 加密 |
| **持仓模型** | 单向（同品种只能持一个方向） | 双向（多空分别记账，可对冲） |
| **订单管理** | Orders + Trades 混合 | Orders → Deals → Positions 三级模型 |
| **回测能力** | 单线程单货币 | 多线程多货币 |
| **经济日历** | 无 | 内置 |
| **编程语言** | MQL4 | MQL5（OOP） |
| **市场地位** | 老牌主流（零售外汇） | 新一代主推 |
| **经纪商授权** | 已停发新授权 | 主推 |

## 2. MT5 业务概念

### 2.1 订单生命周期（Orders → Deals → Positions）

MT5 三级模型，与 MT4 单级 Orders 完全不同：

```
┌─────────┐         ┌─────────┐         ┌──────────┐
│  Order  │ ──执行──> │  Deal   │ ──记账──> │ Position │
│ (订单)  │           │ (成交)  │           │ (持仓)   │
└─────────┘           └─────────┘           └──────────┘
    │                                            │
    │  Pending Order (挂单)                      │
    └── 未成交,等待市场价格触发                   │
                                                │
                                       Close Position
                                            ↓
                                       ┌──────────┐
                                       │  Deal    │
                                       │ (平仓成交)│
                                       └──────────┘
```

**学习要点**：
- **Order**：交易意图（市价单 / 挂单），可能未成交
- **Deal**：实际成交记录（OPEN/CLOSE/IN/OUT）
- **Position**：当前持仓（每个品种一个净持仓，或对冲模式下多空分别）

### 2.2 订单类型

| 类型 | MQL5 常量 | 说明 |
|------|----------|------|
| 市价买 | ORDER_TYPE_BUY | 立即按 Ask 价买入 |
| 市价卖 | ORDER_TYPE_SELL | 立即按 Bid 价卖出 |
| 限价买 | ORDER_TYPE_BUY_LIMIT | 价格跌到指定价时买入 |
| 限价卖 | ORDER_TYPE_SELL_LIMIT | 价格涨到指定价时卖出 |
| 止损买 | ORDER_TYPE_BUY_STOP | 价格涨破指定价时买入 |
| 止损卖 | ORDER_TYPE_SELL_STOP | 价格跌破指定价时卖出 |
| 止限买 | ORDER_TYPE_BUY_STOP_LIMIT | 涨破触发价后,跌到限价才买 |
| 止限卖 | ORDER_TYPE_SELL_STOP_LIMIT | 跌破触发价后,涨到限价才卖 |

### 2.3 持仓模型

```
净持仓模式(Netting):       对冲持仓模式(Hedging):
┌──────────────┐            ┌──────────────┐
│ EURUSD +0.5  │            │ EURUSD +0.3  │ (多头仓)
│ (多单累加)   │            │ EURUSD -0.2  │ (空头仓)
└──────────────┘            └──────────────┘

净持仓 = 0.5                多空独立,可同时持有
同方向再加 0.3 → 净 0.8
反方向减 0.2 → 净 0.6
```

### 2.4 价格与点

| 概念 | 说明 | 示例 |
|------|------|------|
| **Bid** | 买价（你能卖出的价格） | EURUSD Bid=1.10500 |
| **Ask** | 卖价（你能买入的价格） | EURUSD Ask=1.10502 |
| **Spread** | 点差 = Ask - Bid | 2 点（经纪商利润） |
| **Point** | 最小变动单位 | EURUSD Point=0.00001 |
| **Pip** | 1 pip = 10 Point | EURUSD 涨 1 pip = 涨 0.0001 |
| **Digits** | 小数位数 | EURUSD Digits=5 |
| **Volume** | 手数，1 lot = 10万基础货币 | EURUSD 0.1 lot = 1万欧元 |

### 2.5 杠杆与保证金

```
杠杆 1:100, 账户 10000 USD, 买 1 lot EURUSD (1.1050)
  名义金额 = 1.1050 × 100000 = 110500 USD
  保证金 = 110500 / 100 = 1105 USD
  可用保证金 = 10000 - 1105 = 8895 USD
  浮动 1 点 = 1 pip × 100000 × 0.0001 = 10 USD/pip
```

## 3. 实际对接场景

### 3.1 经纪商白标（White Label）

**场景**：金融科技公司想做外汇经纪商，但不想自建 MT5 Server，向大经纪商租用。

**架构**：
```
┌────────────────────────────┐
│ White Label Provider       │
│ (大经纪商,如 Exness)       │
│   ↓ 租用                    │
│ White Label 客户            │
│   ├─ 自有品牌              │
│   ├─ MT5 Server 分租       │
│   └─ Manager 后台         │
└────────────────────────────┘
```

**对接方式**：
- 通过 MT5 Manager API 管理账户、查询交易、设置杠杆
- 通过 Gateway 对接上游流动性

---

### 3.2 PAMM / MAM 跟单系统

**场景**：交易大师操盘，跟随者资金自动跟随操作。

**PAMM（Percentage Allocation Management Module）**：
- 资金按比例分配盈亏
- 大师下单 → 按跟随者资金比例拆分

**MAM（Multi-Account Manager）**：
- 跟随者账户独立持有持仓
- 大师下单 → 复制到每个跟随者账户

**对接架构**：
```
┌──────────────────────────────────┐
│  C# PAMM 服务                    │
│  - 监听主账户下单事件             │
│  - 计算分配比例                  │
│  - 调用 MT5 Manager API 下单     │
└────────────┬─────────────────────┘
             │ MT5 Manager API
             ▼
┌──────────────────────────────────┐
│  MT5 Server                      │
│  ├─ 主账户(操盘大师)              │
│  ├─ 跟随者 A                     │
│  ├─ 跟随者 B                     │
│  └─ 跟随者 C                     │
└──────────────────────────────────┘
```

---

### 3.3 风控监控平台

**场景**：实时监控所有客户持仓，触发风控规则时强制平仓。

**关键监控指标**：
- 净值 / 余额
- 保证金占用率（净值 / 保证金 < 50% 触发预警，< 20% 强平）
- 单品种净持仓
- 大单异常（> 10 lot 立即告警）
- 高频交易检测（1 秒内 > 10 笔）

**对接方式**：
- **主动轮询**：C# 服务定时（1s）调 Manager API 查所有账户
- **事件订阅**：订阅 MT5 Server 交易事件（OnTradeTransaction）

---

### 3.4 自动化做市

**场景**：经纪商在 ECN 模式下做市，提供流动性。

**核心逻辑**：
- 监听订单簿，发现点差过大时挂单
- 接收客户订单后对冲到上游 LP
- 赚取点差 + 对冲残差

**对接方式**：
- C# 做市引擎通过 Manager API 实时下单
- 通过 Gateway 桥接上游 LP

---

### 3.5 数据归档 + 大数据分析

**场景**：把所有 Tick 数据落库，做策略研发、客户画像、监管报送。

**架构**：
```
EA (MT5 Terminal)
   │ ZeroMQ (Tick 流,1M/s)
   ▼
C# 接收服务
   │ Kafka
   ▼
Kafka Topic (tick-stream)
   │
   ├─▶ ClickHouse (实时分析)
   ├─▶ HDFS/S3    (历史归档)
   └─▶ Spark       (大数据计算)
```

## 4. 三种主流对接方式

### 4.1 方式 A：EA + ZeroMQ（推荐用于策略对接）

**适用场景**：交易策略对接、自定义指标、信号转发

**架构**：
```
MT5 Terminal + EA (MQL5)
   │ ZeroMQ REQ
   ▼
C# 服务端
   - 算法计算
   - 风控判断
   - 下单指令
```

**特点**：
- ✅ 无需经纪商授权（EA 在客户端运行）
- ✅ 极低延迟（ZeroMQ 微秒级）
- ✅ 跨语言（C# / Python / Java 都能接）
- ❌ 依赖 MT5 终端常驻运行
- ❌ 无法管理其他用户账户

### 4.2 方式 B：Manager API 直连（企业级）

**适用场景**：PAMM 跟单、风控平台、白标管理

**架构**：
```
C# 服务 (MT5 Manager API)
   │ TCP 私有协议
   ▼
MT5 Server (需经纪商授权 IP)
   ├─ 账户管理
   ├─ 交易查询
   ├─ 强制平仓
   └─ 风控调整
```

**特点**：
- ✅ 直接操作 MT5 Server，权限最大
- ✅ 可管理所有用户账户
- ✅ 无需终端常驻
- ❌ 需经纪商授权服务器 IP
- ❌ 协议私有，需购买 License

### 4.3 方式 C：Gateway 桥接（做市商）

**适用场景**：做市、流动性对接

**架构**：
```
上游 LP（流动性提供商）
   │ FIX 协议
   ▼
MT5 Gateway (插件)
   │
   ▼
MT5 Server
   ├─ 客户订单匹配
   └─ 风险对冲到上游
```

**特点**：
- ✅ 标准化 FIX 协议
- ✅ 多 LP 接入
- ❌ 开发复杂（C++ / DLL）
- ❌ 性能要求极高（< 1ms）

## 5. 代码示例

### 5.1 EA + ZeroMQ 对接（轻量级）

**场景**：EA 每 100ms 把当前 Tick 推送给 C# 服务，C# 返回交易信号。

#### 5.1.1 MQL5 EA 端

```cpp
// SignalClientEa.mq5
// 完整源码见 examples/mt4-mt5/SignalClientEa.md
#property strict
#include <Zmq/Zmq.mqh>

input string ZMQ_ENDPOINT = "tcp://127.0.0.1:5556";  // C# 服务地址
input int    SEND_INTERVAL_MS = 100;                   // 发送间隔

Context   g_context("SignalClient");
Socket    g_req(g_context, ZMQ_REQ);
datetime  g_lastSend = 0;

int OnInit()
{
    // 连接 C# 服务
    if(!g_req.connect(ZMQ_ENDPOINT))
    {
        Print("连接失败: ", ZMQ_ENDPOINT);
        return INIT_FAILED;
    }
    Print("已连接: ", ZMQ_ENDPOINT);
    return INIT_SUCCEEDED;
}

void OnTick()
{
    // 限频:每 100ms 发一次
    if(GetTickCount() - g_lastSend < SEND_INTERVAL_MS) return;
    g_lastSend = GetTickCount();

    // 构造 JSON 请求
    string json = StringFormat(
        "{\"symbol\":\"%s\",\"price\":%.5f,\"time\":\"%s\"}",
        _Symbol, SymbolInfoDouble(_Symbol, SYMBOL_BID),
        TimeToString(TimeCurrent(), TIME_DATE | TIME_SECONDS));

    // 发送请求
    ZmqMsg request(json);
    g_req.send(request);

    // 接收响应
    ZmqMsg reply;
    if(g_req.recv(reply, 1000))  // 1s 超时
        Print("收到: ", reply.getData());
}

void OnDeinit(const int reason)
{
    g_req.disconnect(ZMQ_ENDPOINT);
}
```

### 5.2 MT5 Manager API 直连（企业级）

**场景**：C# 服务直接管理 MT5 账户（PAMM、风控、查询）。

**注意**：MT5 Manager API 是 MetaQuotes 官方提供的 .NET 组件，需要：
1. 购买 MT5 Server License
2. 经纪商授权你的服务器 IP
3. 安装 `MT5ManagerAPI.dll`（NuGet：`MetaTrader5.Manager`）

#### 5.2.1 连接 MT5 Server

```csharp
// examples/MtManagerDemo/Program.cs
using MetaTrader5.Manager;
using MetaTrader5.Manager.Events;

public class Mt5ManagerService
{
    private ManagerAPI _manager;

    public async Task ConnectAsync(string host, int port, string login, string password)
    {
        _manager = new ManagerAPI();

        // 订阅关键事件
        _manager.OnTradeTransaction += HandleTrade;
        _manager.OnUserAdd += HandleUserAdd;
        _manager.OnPing += (_, __) => { /* 心跳 */ };

        // 连接 MT5 Server
        await _manager.ConnectAsync(host, port, login, password);
        Console.WriteLine($"✓ 已连接 MT5 Server: {host}:{port}");
    }

    // 处理交易事件(实时推送,非轮询)
    private void HandleTrade(object sender, TradeTransEventArgs e)
    {
        // e.Trans.TransactionType: TRANSCMD_ADD / TRANS_DEAL_ADD 等
        // e.Trans.OrderId / DealId / Login / Symbol / Volume / Price
        Console.WriteLine(
            $"[交易] user={e.Trans.Login} {e.Trans.Symbol} " +
            $"vol={e.Trans.Volume} price={e.Trans.Price}");
    }

    private void HandleUserAdd(object sender, UserEventArgs e)
    {
        Console.WriteLine($"[用户] 新用户: {e.User.Login} {e.User.Name}");
    }
}
```

#### 5.2.3 强制平仓（风控场景）

```csharp
public class Mt5RiskService
{
    private readonly ManagerAPI _manager;

    // 强制平仓(风控触发)
    public async Task<bool> ClosePositionAsync(ulong positionTicket, string reason)
    {
        try
        {
            // 调用 Manager API 关闭持仓
            await _manager.PositionCloseAsync(positionTicket);
            Console.WriteLine($"[风控] 已强平 {positionTicket}, 原因: {reason}");
            return true;
        }
        catch (Exception ex)
        {
            Console.WriteLine($"[风控] 强平失败 {positionTicket}: {ex.Message}");
            return false;
        }
    }

    // 批量平仓(账户净值跌破阈值时触发)
    public async Task<int> CloseAllPositionsAsync(long login, string reason)
    {
        var positions = await _manager.PositionsRequestAsync(login);
        int closed = 0;
        foreach (var p in positions)
        {
            if (await ClosePositionAsync(p.Ticket, reason))
                closed++;
        }
        return closed;
    }

    // 修改杠杆(风控调整)
    public async Task SetLeverageAsync(long login, int leverage)
    {
        var user = await _manager.UserRequestAsync(login);
        user.Leverage = leverage;
        await _manager.UserUpdateAsync(user);
        Console.WriteLine($"[风控] 已调整 {login} 杠杆为 1:{leverage}");
    }
}
```

#### 5.2.4 PAMM 跟单实现

```csharp
public class PammService
{
    private readonly ManagerAPI _manager;

    // 主账户下单后,按比例复制到跟随者
    public async Task MirrorTradeAsync(
        ulong sourceDealId,
        Dictionary<long, decimal> followers)  // login → 比例
    {
        // 1. 查询主账户成交
        var deal = await _manager.DealRequestAsync(sourceDealId);

        // 2. 按比例复制到每个跟随者
        foreach (var (login, ratio) in followers)
        {
            var volume = Math.Round(deal.Volume * ratio, 2);
            if (volume < 0.01) continue;

            // 调用 Manager API 下单
            var request = new TradeRequest
            {
                Login = login,
                Symbol = deal.Symbol,
                Volume = volume,
                Type = deal.DealType == DEAL_TYPE_BUY
                    ? TRADE_TYPE_BUY
                    : TRADE_TYPE_SELL,
                Comment = $"PAMM-Mirror-{sourceDealId}"
            };

            var result = await _manager.TradeRequestAsync(request);
            Console.WriteLine($"[PAMM] 跟随 {login}: {volume} lot {deal.Symbol} → {result}");
        }
    }
}
```

### 5.3 Tick 数据归档示例

**场景**：通过 EA 接收 Tick 推送到 Kafka，消费端落库 ClickHouse。

```csharp
// examples/MtTickArchiver/TickProducer.cs
using Confluent.Kafka;
using NetMQ;
using NetMQ.Sockets;
using System.Text.Json;

public class TickProducer
{
    public async Task RunAsync()
    {
        // 1. ZeroMQ 接收 EA 推送的 Tick
        using var sub = new SubscriberSocket();
        sub.Connect("tcp://127.0.0.1:5557");
        sub.Subscribe("");  // 订阅所有 Topic

        // 2. Kafka 生产者
        var kafkaConfig = new ProducerConfig
        {
            BootstrapServers = "localhost:9092",
            LingerMs = 5,        // 攒 5ms 批量发送
            BatchSize = 100000
        };
        using var producer = new ProducerBuilder<string, string>(kafkaConfig).Build();

        Console.WriteLine("开始归档 Tick 流...");
        while (true)
        {
            string topic = sub.ReceiveFrameString();
            string body = sub.ReceiveFrameString();

            // 解析 + 推送到 Kafka
            var tick = JsonSerializer.Deserialize<Tick>(body);
            await producer.ProduceAsync("tick-stream",
                new Message<string, string>
                {
                    Key = tick.Symbol,
                    Value = body
                });
        }
    }
}

public record Tick(string Symbol, double Bid, double Ask, DateTime Time);
```

## 6. 部署架构

### 6.1 经纪商标准部署

## 8. 面试话术（10 道 Q&A）

### Q1: MT5 和 MT4 有什么本质区别？

> 三个核心差异：
> 1. **持仓模型**：MT4 单向，同品种只能持一个方向；MT5 双向，可对冲同时持有多空
> 2. **订单模型**：MT4 单级 Orders；MT5 三级 Orders → Deals → Positions，更接近交易所设计
> 3. **资产支持**：MT4 仅外汇+CFD，MT5 全资产（外汇+股票+期货+加密）
>
> 另外 MT5 支持多线程回测、经济日历、OOP 风格的 MQL5。市场地位上 MT4 仍是零售外汇主流，但 MetaQuotes 已停发 MT4 新授权，未来 MT5 是趋势。

### Q2: MT5 三级订单模型（Orders/Deals/Positions）怎么理解？

> 这是 MT5 最容易被混淆的概念：
> - **Order**：交易意图（市价单 / 挂单），可能未成交（挂单未触发）
> - **Deal**：实际成交记录，一个 Order 可能产生 0 个或多个 Deal
> - **Position**：当前持仓，同品种净持仓（净持仓模式）或多空分别（对冲模式）
>
> 例子：你买入 0.5 手 EURUSD，Order 创建 → Deal 成交 → Position 增加 0.5。再加 0.3，第二个 Deal，Position 累加到 0.8。

### Q3: 你们怎么对接 MT5 的？

> 看场景选方式：
> - **策略对接（EA 侧）**：MT5 EA + ZeroMQ，C# 服务做算法和风控。无需经纪商授权，延迟微秒级
> - **企业级管理（PAMM/风控）**：MT5 Manager API 直连 Server，可管理所有账户。需要经纪商授权 IP
> - **流动性接入（做市商）**：MT5 Gateway + FIX 协议对接上游 LP
>
> 我们项目以策略对接为主，EA 推 Tick 到 C#，C# 计算后返回下单信号。

### Q4: Manager API 和 EA 方式有什么区别？

> 4 个维度对比：
>
> | 维度 | Manager API | EA + ZeroMQ |
> |------|-------------|-------------|
> | 位置 | 服务端直连 Server | 客户端在 Terminal |
> | 权限 | 全账户管理 | 仅当前账户 |
> | 依赖 | 需经纪商授权 IP | 仅需 Terminal |
> | 场景 | PAMM/风控/白标 | 策略对接 |
>
> 简单策略用 EA 即可，企业级管理必须 Manager API。

### Q5: PAMM 跟单系统怎么实现？

> 主账户下单后，C# 服务接收事件 → 按跟随者资金比例分配手数 → 调用 Manager API 在跟随者账户下单。
>
> 关键点：
> - **实时性**：主账户成交后 < 100ms 内复制完成
> - **比例分配**：跟随者资金 / 主账户资金 × 主账户手数
> - **风险隔离**：每个跟随者独立账户，强平线独立
> - **幂等性**：同一笔主账户 Deal 只复制一次（用 Deal ID 去重）

### Q6: 风控平台怎么做？

> 三层监控：
> 1. **账户级**：净值回撤 > 20% 预警，> 50% 强平；保证金占用率 < 20% 强平
> 2. **持仓级**：单品种净持仓异常、大单（> 10 lot）告警
> 3. **行为级**：高频交易（1 秒 > 10 笔）、洗盘交易检测
>
> 技术上用 Manager API 订阅 OnTradeTransaction 事件实时推送，配合 Prometheus 监控 + AlertManager 告警。

### Q7: EA + ZeroMQ 对接的关键技术点？

> 三点：
> 1. **mql-zmq 库**：MQL5 端调用 ZeroMQ 的桥接库（libzmq.dll + Zmq.mqh）
> 2. **通信模式**：REQ-REP 同步请求响应，或 PUB-SUB 行情广播
> 3. **协议设计**：JSON 字符串传输，含 symbol/price/seq 字段
>
> 我们 [MtBridge.Server](file:///c:/Users/ZJN/Desktop/jl/project/examples/MtBridge.Server/Program.cs) 用 REP 接收 + Channel 内部管道，Dispatcher fan-out 到风控/统计两个消费者。

### Q8: MT5 部署为什么要选 VPS？

> EA 在 MT5 Terminal 内运行，Terminal 必须常驻。家用电脑断电/断网 = EA 停止 = 策略失效。
>
> VPS 要求：
> - 离 MT5 Server 近（< 10ms 延迟）
> - 7×24 稳定运行
> - Windows Server 系统
>
> 主流选址：纽约 Equinix NY4、伦敦 LD4、阿姆斯特丹 AMS-IX（这些机房和 MT5 Server 同机房）。

### Q9: 净持仓和对冲持仓的区别？

> **净持仓模式**：同品种只能有一个净持仓，新订单累加或冲减。例如已有 0.5 多，再加 0.3 多 = 0.8 多；反手卖 0.2 = 0.6 多。
>
> **对冲持仓模式**：同品种可同时持有多空两个独立持仓。例如已有 0.5 多，再卖 0.3 → 同时有 0.5 多 + 0.3 空，不互相冲销。
>
> 净持仓更接近交易所风格，对冲更适合套利策略。MT5 默认是净持仓，可在账户设置切换为对冲。

### Q10: 做市商和经纪商在 MT5 里有什么区别？

> - **经纪商（Broker）**：提供交易通道给客户，自己不一定承担风险
> - **做市商（MM）**：报出买卖价，承担对手方风险，赚点差
>
> 在 MT5 中：
> - 经纪商用 **Dealer 模式**：客户订单内部撮合，经纪商承担对手方风险
> - 经纪商用 **ECN 模式**：订单直接到上游 LP，经纪商只赚手续费
>
> 经纪商通过 **Gateway** 对接上游 LP，做风险对冲。
