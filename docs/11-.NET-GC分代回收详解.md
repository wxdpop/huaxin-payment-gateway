# .NET GC 分代回收详解

> 本文深入讲解 .NET 垃圾回收器(GC)的分代回收机制,并附可运行的 C# 示例代码,便于学习调试。

## 一、GC 是什么

**垃圾回收器(Garbage Collector)** 是 .NET 运行时自动管理内存的机制。开发者用 `new` 分配内存但不需要手动 `delete`,GC 会在合适的时机自动回收不再使用的对象。

### 为什么需要 GC?

| 痛点(手动管理) | GC 解决 |
|--------------|--------|
| 忘记释放 → 内存泄漏 | 自动跟踪,无引用即回收 |
| 重复释放 → 程序崩溃 | 一个对象只回收一次 |
| 释放后还在用 → 悬空指针 | GC 回收前确保无引用 |
| 多线程同步复杂 | GC 全线程安全 |
| 心智负担重 | 开发者专注业务 |

### GC 的核心目标
1. **快**: 分配内存 O(1),回收尽量不阻塞主线程
2. **省内存**: 及时回收,避免堆膨胀
3. **公平**: 短命对象优先回收(临时对象多)

## 二、分代回收原理

### 2.1 为什么分代?

**核心假设**（弱代假设）:
- **新对象很快变垃圾** (大多数临时变量,方法结束即垃圾)
- **老对象大概率继续存活** (单例、缓存、配置)

不分代的话,每次 GC 都扫整个堆,代价巨大。分代让 GC 只扫小区域(新生代),既快又及时。

### 2.2 三代堆结构

```
┌──────────────────────────────────────────────────────────┐
│                     Managed Heap(托管堆)                  │
├──────────────┬──────────┬──────────┬──────────────────────┤
│   Gen 0      │  Gen 1   │  Gen 2   │  LOH(大对象堆)       │
│   最小       │  中等    │  最大    │  ≥85000 字节         │
│   频繁回收   │  偶尔    │  很少    │  不压缩(只标记清扫)  │
└──────────────┴──────────┴──────────┴──────────────────────┘
       ↑            ↑         ↑
       └────────────┴─────────┘
       对象晋升:Gen0 → Gen1 → Gen2
```

| 代 | 大小预算 | 回收频率 | 包含对象 |
|----|---------|---------|---------|
| **Gen 0** | ~16 MB | 最频繁 | 刚创建的新对象 |
| **Gen 1** | ~16 MB | 中等 | 经过 1 次 GC 仍存活的 Gen0 对象 |
| **Gen 2** | 无上限 | 最少 | 长期存活对象(单例、缓存) |
| **LOH** | - | 与 Gen2 一起 | ≥85000 字节的大对象(数组、字符串) |
| **POH** | - | 与 Gen2 一起 | .NET 5+ 固定对象(供 native 互操作) |

### 2.3 对象生命周期

```
new Object()
   │
   ▼
分配到 Gen 0
   │
   GC 触发(预算超限)
   │
   ▼
标记:GC 扫描所有根(栈变量、静态字段、CPU 寄存器)找可达对象
   │
   ▼
存活?──是──> 晋升到 Gen 1
   │
   否
   │
   ▼
回收(内存归零)
```

**晋升规则**:
- Gen 0 存活 → Gen 1
- Gen 1 存活 → Gen 2
- Gen 2 存活 → 留在 Gen 2(没有 Gen 3)
- 大对象直接进 LOH(不分代)

### 2.4 GC 触发条件

1. **Gen 0 预算超限** - 最常见(每分配 N 字节触发)
2. **Gen 1 预算超限** - 触发 Gen 1+Gen 0
3. **Gen 2 预算超限** - 触发全堆回收(Full GC,最贵)
4. **OS 内存压力** - 系统内存不足
5. **手动调用** - `GC.Collect()` (不推荐生产用)
6. **大对象分配** - LOH 触发 Gen 2 GC

### 2.5 GC 工作流程(三步)

```
1. 标记(Mark)
   ┌────────────────┐
   │ 从根开始 DFS    │  根:栈变量、静态字段、寄存器
   │ 标记所有可达对象 │
   └────────────────┘
        ↓
2. 清扫/压缩(Sweep/Compact)
   ┌──────────────────────────────────┐
   │ 存活对象压缩到一端,腾出连续空间    │
   │ Gen 0/1/LOH: 压缩               │
   │ Gen 2: 标记清扫,不压缩(避免大对象移动) │
   └──────────────────────────────────┘
        ↓
3. 修复指针(Fixup)
   ┌──────────────────────────────────┐
   │ 更新所有指向被移动对象的引用       │
   └──────────────────────────────────┘
```

### 2.6 后台 GC vs 阻塞 GC

| 模式 | 适用代 | 阻塞主线程 | 性能 |
|------|--------|----------|------|
| **并发(后台) GC** | Gen 2 + LOH | 否(后台线程) | 服务器场景推荐 |
| **阻塞 GC** | Gen 0 + Gen 1 | 是(短暂) | 工作站模式 |

.NET Core 3.0+ 默认启用后台 GC,大幅降低 Gen 2 回收对延迟敏感场景的影响。

## 三、示例代码

下面示例演示 GC 行为,可在控制台运行观察日志。

```csharp
// ============================================================================
// GcDemo - .NET GC 分代回收演示
// ============================================================================
// 运行方式:
//   dotnet run --project GcDemo
//   或: dotnet run -c Release  (Release 模式观察实际性能)
//
// 观察重点:
//   1. 对象分配到 Gen 0
//   2. Gen 0 满了 → 触发 GC,存活对象晋升 Gen 1
//   3. Gen 1 满了 → 触发 GC,存活对象晋升 Gen 2
//   4. 大对象直接进 LOH
// ============================================================================

using System;
using System.Runtime;
using System.Threading;

Console.WriteLine("=== .NET GC 分代回收演示 ===\n");
Console.WriteLine($"GC 服务器模式: {System.Runtime.GCSettings.IsServerGC}");
Console.WriteLine($"GC 延迟模式: {System.Runtime.GCSettings.LatencyMode}");
Console.WriteLine($"当前堆内存: {GC.GetTotalMemory(false) / 1024} KB\n");

// =====================================================================
// Demo 1: 新对象分配到 Gen 0
// =====================================================================
Console.WriteLine("--- Demo 1: 新对象分配到 Gen 0 ---");
var obj1 = new { Name = "测试对象1", Data = new byte[1000] };
Console.WriteLine($"obj1 在 Gen: {GC.GetGeneration(obj1)}");  // 期望: 0

// =====================================================================
// Demo 2: 触发 Gen 0 GC,存活对象晋升 Gen 1
// =====================================================================
Console.WriteLine("\n--- Demo 2: 触发 Gen 0 GC,存活对象晋升 Gen 1 ---");
Console.WriteLine($"触发前 - obj1 在 Gen: {GC.GetGeneration(obj1)}");
Console.WriteLine($"触发前 - Gen 0 回收次数: {GC.CollectionCount(0)}");
Console.WriteLine($"触发前 - Gen 1 回收次数: {GC.CollectionCount(1)}");

// 分配一批临时对象,触发 Gen 0 GC
for (int i = 0; i < 100000; i++)
{
    var tmp = new byte[100];
}
Console.WriteLine($"分配 100K 临时对象后 - obj1 在 Gen: {GC.GetGeneration(obj1)}");
Console.WriteLine($"Gen 0 回收次数: {GC.CollectionCount(0)}");
Console.WriteLine($"Gen 1 回收次数: {GC.CollectionCount(1)}");

// =====================================================================
// Demo 3: 大对象进 LOH
// =====================================================================
Console.WriteLine("\n--- Demo 3: 大对象(≥85000 字节)直接进 LOH ---");
var bigObj = new byte[100_000];  // 100KB,超过 85000 字节阈值
Console.WriteLine($"bigObj (100KB) 在 Gen: {GC.GetGeneration(bigObj)}");  // 期望: 2 (LOH)

// =====================================================================
// Demo 4: 手动触发 GC,观察代际变化
// =====================================================================
Console.WriteLine("\n--- Demo 4: 手动触发 GC ---");
Console.WriteLine($"触发前 - obj1 在 Gen: {GC.GetGeneration(obj1)}, bigObj 在 Gen: {GC.GetGeneration(bigObj)}");

GC.Collect(0);  // 只回收 Gen 0
Console.WriteLine($"GC.Collect(0) 后 - obj1 在 Gen: {GC.GetGeneration(obj1)}, bigObj 在 Gen: {GC.GetGeneration(bigObj)}");

GC.Collect(1);  // 回收 Gen 0 + Gen 1
Console.WriteLine($"GC.Collect(1) 后 - obj1 在 Gen: {GC.GetGeneration(obj1)}, bigObj 在 Gen: {GC.GetGeneration(bigObj)}");

GC.Collect(2);  // 全堆回收(Gen 0/1/2 + LOH)
Console.WriteLine($"GC.Collect(2) 后 - obj1 在 Gen: {GC.GetGeneration(obj1)}, bigObj 在 Gen: {GC.GetGeneration(bigObj)}");

// =====================================================================
// Demo 5: 大对象 LOH 碎片化演示
// =====================================================================
Console.WriteLine("\n--- Demo 5: LOH 碎片化(避免大对象频繁分配) ---");
Console.WriteLine("持续分配大对象会快速膨胀 LOH,即使被释放也可能不压缩...");
var bigList = new System.Collections.Generic.List<byte[]>();
for (int i = 0; i < 5; i++)
{
    bigList.Add(new byte[100_000]);  // 5 × 100KB = 500KB LOH 占用
    Console.WriteLine($"分配第 {i + 1} 个 100KB,堆内存: {GC.GetTotalMemory(false) / 1024} KB");
}

bigList.Clear();  // 释放引用,但 LOH 不立即压缩
GC.Collect();
Console.WriteLine($"释放后 GC.Collect(),堆内存: {GC.GetTotalMemory(false) / 1024} KB (可能不会降到 0)");

// =====================================================================
// Demo 6: 弱引用(WeakReference)演示
// =====================================================================
Console.WriteLine("\n--- Demo 6: WeakReference 弱引用不阻止 GC ---");
var weakRef = new WeakReference<string>(new string('x', 1000));
Console.WriteLine($"弱引用目标存在: {weakRef.TryGetTarget(out _)}");

GC.Collect();
Console.WriteLine($"GC 后弱引用目标还存在: {weakRef.TryGetTarget(out _)}");  // 期望: False(已回收)

// =====================================================================
// Demo 7: IDisposable + Finalizer 模式
// =====================================================================
Console.WriteLine("\n--- Demo 7: IDisposable + Finalizer 模式 ---");
Console.WriteLine("推荐: 显式 Dispose,避免 Finalizer 开销(GC 需要两次回收)");
using (var resource = new NativeResource())
{
    Console.WriteLine($"使用 NativeResource...");
}  // 自动调用 Dispose,立即释放
Console.WriteLine($"已 Dispose,无需等待 Finalizer");
Thread.Sleep(100);

// =====================================================================
// Demo 8: GC 暂停模式(低延迟场景)
// =====================================================================
Console.WriteLine("\n--- Demo 8: GC 延迟模式(关键路径用) ---");
Console.WriteLine($"原延迟模式: {System.Runtime.GCSettings.LatencyMode}");

// 在关键路径降低 GC 侵略性
var old = System.Runtime.GCSettings.LatencyMode;
System.Runtime.GCSettings.LatencyMode = GCLatencyMode.SustainedLowLatency;
try
{
    Console.WriteLine("进入低延迟区域...");
    for (int i = 0; i < 1000000; i++)
    {
        var tmp = new byte[100];
        // GC 会尽量避免触发,降低延迟抖动
    }
}
finally
{
    System.Runtime.GCSettings.LatencyMode = old;
}
Console.WriteLine("退出低延迟区域");

Console.WriteLine("\n=== 演示结束 ===");

// ============================================================================
// NativeResource - 演示 IDisposable + Finalizer 模式
// ============================================================================
public class NativeResource : IDisposable
{
    private bool _disposed;

    public NativeResource()
    {
        Console.WriteLine($"[NativeResource] 构造");
    }

    public void Dispose()
    {
        Console.WriteLine($"[NativeResource] Dispose 释放");
        Dispose(true);
        GC.SuppressFinalize(this);  // ★ 关键: Dispose 后不再触发 Finalizer
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;
        if (disposing)
        {
            // 释放托管资源
        }
        // 释放非托管资源(native 句柄等)
        _disposed = true;
    }

    ~NativeResource()
    {
        Console.WriteLine($"[NativeResource] Finalizer 触发(说明 Dispose 未被调用,GC 兜底)");
        Dispose(false);
    }
}
```

## 四、性能调优

### 4.1 减少内存分配

```csharp
// ❌ 每次循环分配新字符串(频繁触发 GC)
string result = "";
for (int i = 0; i < 1000; i++)
    result += i.ToString();

// ✅ 用 StringBuilder(可重用缓冲区,减少分配)
var sb = new StringBuilder();
for (int i = 0; i < 1000; i++)
    sb.Append(i);
var result = sb.ToString();

// ✅ 用 Span<T>/stackalloc 避免堆分配(小内存)
Span<byte> buffer = stackalloc byte[100];  // 在栈上分配,不进堆
```

### 4.2 对象池化(ArrayPool)

```csharp
using System.Buffers;

// ❌ 每次都 new,触发 LOH GC
byte[] buffer = new byte[100_000];
ProcessData(buffer);

// ✅ 从池租用,用完归还,零 GC 压力
byte[] buffer = ArrayPool<byte>.Shared.Rent(100_000);
try
{
    ProcessData(buffer);
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer);
}
```

### 4.3 大对象避免 LOH 碎片

- **LOH 不压缩** - 大对象频繁分配释放会导致碎片
- **阈值 85000 字节** - 用 `ArrayPool` 或拆分多个小对象避开
- **切片处理** - 大数据流式处理,避免整体加载

### 4.4 GC 设置调优

```csharp
// 启动时配置(在 Program.cs 顶部)
System.Runtime.GCSettings.LatencyMode = GCLatencyMode.Batch;  // 吞吐优先
// 或: GCLatencyMode.Interactive  // 平衡
// 或: GCLatencyMode.SustainedLowLatency  // 延迟优先(.NET 4.7+)
// 或: GCLatencyMode.NoGCRegion  // 关键路径用,限时无 GC
```

```xml
<!-- .runtimeconfig.json -->
{
  "configProperties": {
    "System.GC.Server": true,            // 启用服务器 GC(高并发推荐)
    "System.GC.Concurrent": true,        // 启用后台 GC(Gen 2 并发)
    "System.GC.HeapCount": "16",          // 服务器 GC 堆数(默认=CPU 核心)
    "System.GC.LOHCompactionMode": "1"    // 强制 LOH 压缩
  }
}
```

### 4.5 诊断工具

```bash
# 命令行

dotnet-trace collect -p <pid>               # CPU/GC 追踪

dotnet-dump collect -p <pid>                # 完整 dump

```

## 五、面试话术

### Q1: .NET GC 是怎么分代的?为什么分代?

> **三代(Gen 0/1/2)+ LOH(大对象堆)**。
>
> 分代基于弱代假设:新对象很快变垃圾,老对象大概率继续存活。所以 Gen 0 频繁回收(只扫小区域,快),Gen 2 几乎不动。每次 GC 后,存活对象晋升一代。这样既保证及时回收,又避免每次扫全堆的代价。

### Q2: LOH 是什么?为什么单独处理?

> LOH(Large Object Heap)存储 ≥85000 字节的大对象(数组、字符串)。它单独处理有两个原因:
> 1. **不压缩** - 大对象移动成本太高,只标记清扫,不压缩。所以 LOH 容易碎片化。
> 2. **直接当 Gen 2 算** - 因为大对象分配代价高,直接当老对象,触发 Gen 2 GC 才回收。
>
> 优化:用 `ArrayPool<T>` 池化大数组,避免 LOH 碎片。

### Q3: GC 触发的时机是什么?

> 主要是 **预算超限**:每代有预算(如 Gen 0 默认 16MB),分配超出预算就触发 GC。
> - Gen 0 满 → 触发 Gen 0 回收(最常见)
> - Gen 1 满 → 触发 Gen 1 + Gen 0 回收
> - Gen 2 满 → 全堆回收(最贵)
> - LOH 分配 → 算作 Gen 2,触发 Gen 2 GC
>
> 还有手动 `GC.Collect()` (不推荐) 和 OS 内存压力触发。

### Q4: 后台 GC 是什么?为什么需要?

> .NET Core 3.0+ 默认启用后台 GC。Gen 2 + LOH 的回收在后台线程进行,不阻塞主线程。
>
> 之前 Gen 2 GC 是 Stop-The-World,暂停所有应用线程,可能卡顿几百毫秒。后台 GC 让 Gen 2 回收和应用线程并发执行,大幅降低延迟抖动,对延迟敏感场景(交易、Web)非常关键。

### Q5: 如何避免 GC 压力?

> 五个手段:
> 1. **减少分配** - 用 StringBuilder 替代字符串拼接,Span 替代 byte[]
> 2. **对象池化** - ArrayPool / ObjectPool 租用归还,零 GC
> 3. **结构体替代类** - 值类型在栈上,不进堆
> 4. **避免 LOH** - 拆分大对象,避开 85000 字节阈值
> 5. **延迟模式** - 关键路径用 `SustainedLowLatency` 降低 GC 侵略性

### Q6: Dispose 和 Finalizer 的关系?

> **推荐 Dispose,慎用 Finalizer**。
>
> Finalizer 是 GC 兜底机制,但有两个代价:
> 1. **需要两次 GC** - 第一次 GC 把对象放入 Freachable 队列,Finalizer 线程执行后,下次 GC 才真正回收
> 2. **延长对象生命周期** - 即使没引用,也要等 Finalizer 执行完
>
> 正确模式是 **Dispose + Finalizer**:
> ```csharp
> public void Dispose()
> {
>     Dispose(true);
>     GC.SuppressFinalize(this);  // ★ Dispose 后跳过 Finalizer
> }
> ```
> 显式调用 Dispose 时,`SuppressFinalize` 让 GC 跳过 Finalizer,直接回收。

### Q7: 弱引用(WeakReference)是什么?有什么用?

> 弱引用**不阻止 GC 回收**,但 GC 没回收时还能拿到对象。用途是缓存:缓存的对象如果还在内存就用,被 GC 回收了就重新创建。
>
> ```csharp
> var weak = new WeakReference<string>(new string('x', 1000));
> if (weak.TryGetTarget(out var s)) Console.WriteLine("还在");
> else Console.WriteLine("已被 GC 回收,需重建");
> ```
> 大型缓存系统常用 `WeakReference<T>` 避免缓存膨胀。

### Q8: 服务器 GC 和工作站 GC 有什么区别?

> **工作站 GC**: 单堆,适合客户端应用(桌面、低并发)
> **服务器 GC**: 多堆(每 CPU 一个堆),适合高并发服务端
>
> .NET Core 默认根据环境自动选择。可在 .runtimeconfig.json 配置:
> ```json
> { "System.GC.Server": true }
> ```
>
> ASP.NET Core 服务端默认服务器 GC,吞吐优先,延迟略高但整体性能更好。

### Q9: 怎么诊断 GC 问题?

> 三步:
> 1. **dotnet-counters monitor** - 看 `gen-0-gc-count`、`gen-2-gc-count`、`gen-2-heap-size`
> 2. **dotnet-gcdump** - 拿堆快照,看哪个类型的对象最多
> 3. **Visual Studio 性能探查器** - .NET 对象分配跟踪,看分配热点
>
> 现象判断:
> - Gen 2 频繁回收 → 大对象或老对象过多
> - LOH 碎片 → 大数组频繁分配
> - GC 暂停明显 → 改用后台 GC

### Q10: 你的项目怎么优化 GC?

> 三个层次:
> 1. **代码层**: ArrayPool 池化 Kafka 消息缓冲,Span 处理协议解析,避免字符串拼接
> 2. **配置层**: 启用服务器 GC + 后台 GC + LOH 压缩
> 3. **架构层**: 用对象池替代 Singleton 缓存(避免长期占用 Gen 2),分批处理大数据避免整体加载
>
> 我们支付网关里 Kafka 消费者每次处理消息用 ArrayPool 租缓冲,千万级吞吐也几乎零 GC。
